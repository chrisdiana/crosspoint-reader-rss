SDK Surface

The firmware installs these globals before your script is evaluated: renderer, input, storage, http, app, button constants, and font constants.

Elk syntax: use let for variables and lifecycle hooks. This firmware build does not implement var, and app examples avoid const so browser-tested code stays close to device syntax.
Preview limit: this page uses the browser JavaScript engine with SDK mocks. It is useful for layout and input iteration, but it is not Elk. The validator blocks known Elk-incompatible patterns, yet final scripts still need an Elk/device run for parser edge cases.
rendererDraw into the e-ink framebuffer and request refreshes. inputRead edge-triggered and held button state. storageRead and write text files and numeric values on the SD card. httpFetch text over the network with blocking GET requests. appLog to serial and exit the app.

App Files

Scripts live on the SD card under /.crosspoint/apps/*.js. The home menu scans that folder and launches each file as an app.

The browser editor enforces this app format before running: include @name, include a supported @icon, use let lifecycle hooks, and avoid var.
// @name Hello World
// @icon File
  • @name sets the home menu label. Missing names fall back to the filename without .js.
  • @icon accepts Folder, Text, Image, Book, File, Recent, Settings, Transfer, Library, Wifi, Hotspot, or Bookmark. Unknown icons fall back to File.

Lifecycle

Declare lifecycle hooks with assignment syntax. This matches the firmware's Elk parser path and keeps browser emulation close to device behavior.

All examples intentionally use let. Do not rewrite them to var; var is unsupported by this Elk build.
let onEnter = function() {
  renderer.clear();
  renderer.text(FONT_HEADING, 40, 80, "Ready");
  renderer.flush();
};

let loop = function() {
  if (input.pressed(BTN_BACK)) app.finish();
};

let onExit = function() {
  app.log("closed");
};
  • onEnter runs once after the script loads.
  • loop runs repeatedly for input and state updates.
  • onExit runs when the activity is popped, including after app.finish().

Renderer

renderer.clear()Clear the framebuffer to white. renderer.text(font, x, y, text[, black])Draw text at portrait coordinates. black defaults to true. renderer.fillRect(x, y, w, h[, black])Draw a filled rectangle. black=false draws white. renderer.rect(x, y, w, h)Draw a one-pixel outline rectangle. renderer.width()Current logical screen width. X3 is 528, X4 is 480 in portrait. renderer.height()Current logical screen height. X3 is 792, X4 is 800 in portrait. renderer.flush()Request an e-ink refresh for the changed framebuffer.

Only call flush() after changing pixels. Calling it every loop() iteration causes repeated X3_DRF refreshes, flicker, ghosting, and delayed input response.

Fonts

FONT_SMALLSmall UI/help text. FONT_UIDefault UI label text. FONT_BODYReadable body copy. FONT_HEADINGLarge heading text.

Input

input.pressed(button)true for the loop tick where the button became pressed. input.released(button)true for the loop tick where the button was released. input.held(button)true while the button is currently down.

Button constants: BTN_BACK, BTN_OK, BTN_LEFT, BTN_RIGHT, BTN_UP, and BTN_DOWN.

Use pressed() for menu navigation and one-shot actions. Use held() for continuous movement. Always handle BTN_BACK so the app can exit quickly.

let selected = 0;
let dirty = true;

let draw = function() {
  renderer.clear();
  renderer.text(FONT_UI, 32, 60, selected);
  renderer.text(FONT_SMALL, 32, 110, "UP/DOWN changes, OK selects");
  renderer.flush();
};

let loop = function() {
  if (input.pressed(BTN_BACK)) {
    app.finish();
    return;
  }
  if (input.pressed(BTN_UP) && selected > 0) {
    selected = selected - 1;
    dirty = true;
  }
  if (input.pressed(BTN_DOWN)) {
    selected = selected + 1;
    dirty = true;
  }
  if (input.pressed(BTN_OK)) {
    app.log(selected);
  }
  if (dirty) {
    dirty = false;
    draw();
  }
};

App

app.log(value)Write a value to serial logs. The emulator mirrors it in Console. app.finish()Request native activity exit. onExit will run before the app is destroyed.

Storage

storage.read(path)Read a text file from the SD card. Returns a string or null. Empty files currently also read as null. storage.readNumber(path[, fallback])Read a numeric value from a text file. Returns fallback, or 0, when missing or invalid. storage.write(path, data)Write data to the SD card as text. Numbers are stringified by the native binding. 
let count = 0;

let onEnter = function() {
  count = storage.readNumber("/.crosspoint/apps/count.txt", 0);
};

let loop = function() {
  if (input.pressed(BTN_OK)) {
    count = count + 1;
    storage.write("/.crosspoint/apps/count.txt", count);
  }
};

HTTP

http.get(url)Fetch a URL and return response text or null on failure. Firmware calls are blocking.

Use HTTP sparingly in loop(); long network calls block input and rendering while they run.

For apps that fetch once, do the request from onEnter, cache the result in memory or storage, then draw once. For user-triggered refresh, set a loading message, flush, perform the request, then redraw.

let body = "Loading...";

let draw = function() {
  renderer.clear();
  renderer.text(FONT_HEADING, 28, 50, "HTTP GET");
  renderer.text(FONT_SMALL, 28, 105, body);
  renderer.text(FONT_SMALL, 28, renderer.height() - 50, "BACK exits, OK reloads");
  renderer.flush();
};

let fetchNow = function() {
  body = "Loading...";
  draw();
  let response = http.get("https://example.com/status.txt");
  if (response) {
    body = response;
    storage.write("/.crosspoint/apps/status-cache.txt", response);
  } else {
    body = "Request failed";
  }
  draw();
};

let onEnter = function() {
  let cached = storage.read("/.crosspoint/apps/status-cache.txt");
  if (cached) body = cached;
  draw();
};

let loop = function() {
  if (input.pressed(BTN_BACK)) app.finish();
  if (input.pressed(BTN_OK)) fetchNow();
};

Complete Minimal App

This app uses drawing, button input, SD-card persistence, logging, and clean exit without refreshing every frame.

// @name Counter
// @icon Text

let count = 0;
let dirty = true;

let draw = function() {
  renderer.clear();
  renderer.text(FONT_HEADING, 42, 80, "Counter");
  renderer.text(FONT_BODY, 42, 145, count);
  renderer.text(FONT_SMALL, 42, 220, "OK +1, LEFT reset, BACK exit");
  renderer.flush();
};

let onEnter = function() {
  count = storage.readNumber("/.crosspoint/apps/counter.txt", 0);
  dirty = true;
};

let loop = function() {
  if (input.pressed(BTN_BACK)) {
    storage.write("/.crosspoint/apps/counter.txt", count);
    app.finish();
    return;
  }
  if (input.pressed(BTN_OK)) {
    count = count + 1;
    app.log(count);
    dirty = true;
  }
  if (input.pressed(BTN_LEFT)) {
    count = 0;
    dirty = true;
  }
  if (dirty) {
    dirty = false;
    draw();
  }
};

Redraw Pattern

For static screens, draw and flush in onEnter. For interactive screens, redraw only when input changes visible state.

On device, repeated large redraws can expose Elk parser/runtime limits that the browser will not reproduce. For complex screens, draw the stable background once and redraw only the changed cells, rows, or labels.

let selected = 0;

let draw = function() {
  renderer.clear();
  renderer.text(FONT_UI, 30, 60, selected);
  renderer.flush();
};

let onEnter = function() {
  draw();
};

let loop = function() {
  if (input.pressed(BTN_BACK)) {
    app.finish();
    return;
  }
  if (input.pressed(BTN_DOWN)) {
    selected = selected + 1;
    draw();
  }
};

Elk Compatibility

  • Elk is a small JavaScript engine, not a full browser runtime. Avoid DOM APIs, timers, promises, modules, classes, and large libraries.
  • Use let for variables. This Elk build does not implement var; avoid const in app scripts.
  • Do not use browser constructors like String() or Number(). Native SDK calls stringify values where supported.
  • Avoid == null, != null, === null, and !== null. For SDK calls that return string-or-null, use positive truthy checks such as if (value).
  • Prefer simple functions, numbers, strings, and scalar variables. Avoid arrays for now; this Elk build does not support normal browser-style array behavior.
  • Keep scripts small. Deeply nested code can hit parser stack limits on device before it fails in the browser emulator.
  • Avoid while loops. This Elk build reports 'while' not implemented; use bounded for loops only where necessary.
  • Avoid repeated full redraws in loop() for complex screens. Use dirty rectangles or small redraw helpers for changed regions.
